经验总结（更新中）

Created2020-08-19Updated2021-01-20Visited

在这里总结一些踩过的坑以及一些小技巧。

  • 容易犯错的地方（fst原因大全）
    1. 数组大小有时候会是 $2\times 10^5$ 甚至是 $5\times 10^5$，不要想当然直接开 maxn=1e5+9，运气好的话在pretest中RE了还能查出来，要是运气不好过了pretest那就等着fst吧
    2. 初始化的时候尽量写 maxx=-inf,minn=inf 这样的极端数据，无脑初始化为 0 是一个坏习惯
    3. 不开 long long 见祖宗，被坑了无数次了
    4. double 的精度并没有想象的那么高，用的时候要小心
    5. 不要用 std::fill 初始化二维数组，还不如 memset，实在不行就循环赋值

  • 小优化

    1. 读01矩阵这样的东西用 scanf("%c",&x) 比用 scanf("%1d",&x) 稍微快一点
  • 一些经验
    1. 指针和宏定义稍有不慎就会出奇怪的毛病且非常难以调试，用不好就不要用
    2. 不要吃饱了撑着写一堆 register 变量或者位运算，有时候编译器自己会优化的，多用 register 并不一定使程序变快，甚至可能会负优化；多用位运算也不一定会变快，出错的概率倒是大大增加
    3. 想不出做法的时候试试暴力，有奇效
    4. 不只是序列可以分块做，图也是可以分块的
    5. 单调栈非常有用（被这东西坑了两次了）
    6. 如果觉得需要使用一些比较复杂的数据结构，最好再分析一遍题意，否则很可能把简单问题复杂化

  • 容易忘记的小知识点

    1. 斐波那契数列通项公式 $\large f_n=\dfrac{(\frac{1+\sqrt 5}{2})^n-(\frac{1-\sqrt 5}{2})^n}{\sqrt 5}$

    2. 在只含加法和乘法的式子中，如果最后的运算结果需要对p取模，那么可以在运算过程中随便取模

  • 其他
    1. 能不用Java就不用Java，标准输入输出慢的要死，即便用快读快写耗时依然是 scanf 的 $10$ 倍！并且Java的快读快写非常难写。
